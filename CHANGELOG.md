## [Unreleased]

## [0.0.1-rc.3] - 2026-04-25

### Fixed
- Daemon crashed on startup with `Error: Can't find meta/_journal.json file` whenever `flockctl` was launched from any directory other than the package root (i.e. essentially always after `npm install -g flockctl`). The drizzle migrator was being passed a literal `"./migrations"`, which it resolved against `process.cwd()` instead of the package root. `src/db/migrate.ts` now resolves `migrations/` relative to its own module URL, so it works in both the dev source layout (`src/db/migrate.ts → ../../migrations`) and the published `dist/` layout (`dist/db/migrate.js → ../../migrations`). The smoke-test harness (`tests/smoke/_harness.ts`) was hiding this regression because it spawned the daemon with `cwd=repoRoot`; it now spawns from the isolated `FLOCKCTL_HOME` so any future cwd-relative path bug fails the smoke tier before it ships.

## [0.0.1-rc.2] - 2026-04-25

### Breaking
- Remote servers now require SSH. Existing direct-HTTP remote-server entries are automatically purged on first boot of this version — Flockctl logs the names it removed. Re-add them over SSH via the UI or `flockctl remote-add`. See `docs/REMOTE-ACCESS.md`.
- AGENTS.md layering compressed from five layers to three. The private `.flockctl/AGENTS.md` files at both workspace and project scope have been retired; each scope now owns exactly one editable `AGENTS.md` at its root. Agent sessions are prompted from `user` → `workspace-public` → `project-public` only. `GET /projects/:id/agents-md` returns `{ layers: { "project-public": LayerContent } }` and `PUT /projects/:id/agents-md` body is `{ content: string }` — the `layer` selector is gone. Same shape change for the workspace endpoints. `GET /projects/:id/agents-md/effective` resolves three layers; `GET /workspaces/:id/agents-md/effective` resolves two.
- Removed the legacy YAML → JSON config migration in `loadWorkspaceConfig` / `loadProjectConfig`; `.flockctl/config.yaml` files are no longer auto-migrated and must be converted to `config.json` manually.
- Removed the legacy Claude Code plugin cleanup run on daemon startup (`src/services/plugin-cleanup.ts`).
- Removed Windows `git config core.symlinks` guidance from `README.md` and `docs/AGENTS-LAYERING.md`. Symlink creation in the runtime still works; only the user-facing docs bullet is gone.

### Added
- `flockctl remote-bootstrap` CLI — one-shot setup for remote access. Ensures the daemon is running, mints a labelled remote-access token, and (with `--print-token`) emits exactly one line on stdout for scripting. Stdout is otherwise silent; errors go to stderr with token-redaction. Flags: `--print-token`, `-l/--label <name>`, `-p/--port <number>`.
- SSH-based remote server support. New `src/services/ssh-tunnels/` module owns the lifecycle of ssh child processes that forward a local port to a remote Flockctl daemon, with `BatchMode=yes`, classified stderr error codes, ready-gate probing, and reconnect handling. New `tests/cli-docker/scenarios/ssh-remote-full-stack.spec.ts` runs the full remote stack against a real `sshd` container.
- Update notifications. Sidebar footer shows the daemon version with a one-click "Update" button that runs the right `npm install` based on install mode (global vs. local dep). Backed by `GET /meta/version` (`{ current, latest, updateAvailable, installMode }` against the npm registry), `POST /meta/update` (fire-and-forget, returns `202`, refuses with `400` for unknown install mode, `409` if another install is already in flight), and `GET /meta/update` for polling worker state (`idle | running | success | error`).
- Pending permissions endpoints — `GET /chats/:id/pending-permissions` and `GET /tasks/:id/pending-permissions` retrieve open permission requests for a chat or task session. The UI re-hydrates pending requests on mount so the permission-request card no longer disappears after a page reload while a task or chat is blocked on a tool approval.
- Multi-layer AGENTS.md guidance loader and injection. The merged `user → workspace-public → project-public` content is now resolved at session start and injected into both task and chat agent system prompts via a single `loadAgentGuidance` seam. `GET /projects/:id/agents-md/effective` and `GET /workspaces/:id/agents-md/effective` return the exact layered merge (per-layer `layers[]`, `totalBytes`, `truncatedLayers`, `mergedWithHeaders`) the agent will see.
- AGENTS.md editor in project and workspace settings. A dedicated CodeMirror editor and save button for the scope's single `AGENTS.md`; a read-only "Effective preview" accordion below shows the merged `mergedWithHeaders` view. Byte counter turns yellow at 200 KiB and red at 256 KiB; Save is hard-blocked above the 256 KiB backend limit. Save state is announced via aria-live, errors surface as toasts.
- `flockctl agents show <path> [--effective | --layers]` — inspect merged AGENTS.md guidance for a project or workspace path. Defaults to project mode; pass `--workspace` for workspace scope. `--layers` prints a JSON summary (per-layer byte sizes, paths, truncation flags) instead of the merged string. Pipe-friendly (no ANSI escapes); errors / warnings go to stderr. A path is "known" if it contains a `.flockctl/` scaffold on disk — no daemon required.
- Chat approvals — symmetric with the task-side flow. `POST /chats` and `PATCH /chats/:id` accept `requiresApproval: boolean`; when set, a chat flips `approval_status` to `pending` after every successful assistant turn and surfaces as a `chat_approval` blocker in `GET /attention` (alongside `task_approval`). Clear via `POST /chats/:id/approve` and `POST /chats/:id/reject` (optional `{ note }`); both fire `attention_changed` so the Inbox refreshes.
- Per-chat adaptive thinking toggle and reasoning effort picker. New "Brain" control in the chat composer toolbar exposes "Enable adaptive thinking" (off skips the thinking step entirely) and an effort selector (`low/medium/high/max`, default `high`). Persisted per chat via `thinking_enabled` / `effort` columns; surfaced on `GET /chats/:id`, accepted on `PATCH /chats/:id` (`thinkingEnabled`, `effort`), and overridable per turn on `POST /chats/:id/messages` and `POST /chats/:id/messages/stream`. Existing chats upgrade with `thinking_enabled = true` and `effort = NULL` (byte-identical until opt-in).
- Per-agent attribution for TodoWrite snapshots. New `chat_todos.parent_tool_use_id` column (`NULL` for the main agent, `toolu_…` for sub-agents spawned via the Claude Agent SDK's `Task` tool) plus index `(chat_id, parent_tool_use_id, created_at DESC)`. The "Todo history" drawer pivots its previously-flat feed into one tab per agent (main + each sub-agent), each tab showing its own latest snapshot and history.
- Chat composer follow-up queue. Pressing Send while a response streams appends the new message to a queued-messages bar above the composer instead of starting a second concurrent turn; the bar shows each queued prompt with a ✕ to un-queue it before it runs, and the next item drains automatically as soon as the current turn ends. Pressing Stop aborts only the current turn — queued messages keep running.
- Per-chat draft preservation. Each chat keeps its own composer draft (text + pending attachments) until you send or close the tab — switching chats no longer wipes unsent input.
- Mission Control project layout with Board / Tree / Swimlane view modes (`?view=` URL state). Project detail page redesigned as a tabbed surface (`?tab=`): **Plan** (always renders the 3-column Mission Control board), **Runs** (analytics + flat per-project tasks table), **Templates & Schedules**, **Config** (folds in General, AI Configuration, Execution, Environment Variables, Gitignore, AGENTS.md, Skills, MCP, Secrets, Danger Zone — no "Full settings" hop). Legacy `/projects/:id/settings` URL redirects to `?tab=config`. Workspace detail page aligned with the same tabbed layout (Plan / Runs / Templates & Schedules / Config) plus MissionControl KPI bar and unified Chat button.
- Cross-project Kanban for the Tasks page (`?view=kanban`) — tasks from every project laid out in status swim-lanes (Queued, Running, Pending Approval, Done, Failed). Each card shows the project, AI key label, and the model that actually ran. New `actual_model_used` field on `GET /tasks` and `GET /tasks/:id` (most recent model from `usage_records`, surfaced in the table and Kanban cards instead of "Default").
- Milestone detail panel in the Board view right rail. Clicking a milestone in the tree now surfaces title, status, vision, aggregate slice/task progress, and summed duration + cost. Includes a **Run** / **Re-run** button for milestone-level auto-execution (disabled while a run is already in flight). `GET /projects/:id/tree` enriches each plan task's `summary` with `duration_ms` and `total_cost_usd`.
- Tasks list "Show superseded" toggle (and `include_superseded=true` on `GET /tasks`) plus a "Build after re-run" stat card counting successful runs that only landed because they were a re-run. Backed by new fields on `GET /tasks/stats` (`supersededFailures`, `buildAfterRerun`) and `GET /metrics/overview` (`productivity.buildAfterRerun`, `productivity.supersededFailures`, `productivity.effectiveSuccessRate`).
- Rerun chain on the task detail page. When a task was produced by a rerun, the page shows "Rerun of #..." linking back to the original; any reruns spawned from the task appear in a "Re-runs" list with their status. `GET /tasks/:id` returns a `children` array of direct rerun descendants (manual `/rerun` + auto-retry) plus the existing `parentTaskId` to climb upward. `GET /tasks/stats` adds `failedRerun` / `failedNotRerun` (failed tasks that were vs. weren't rerun).
- Chat "Changes" card. After an assistant turn that touched files, the chat transcript ends with a summary card (`N file(s) changed, +X / -Y`) plus a **Show Diff** toggle that expands a unified diff via the shared `InlineDiff` component — identical UX to the task change-set card. Powered by `GET /chats/:id/diff` (mirrors `GET /tasks/:id/diff` shape) and the `chat_diff_updated` WebSocket event, both delivered live as the agent edits files.
- Inline code-diff viewer (`InlineDiff`) shared between the task change-set card and the chat transcript. Unified-diff input is parsed into per-file collapsible cards with `+N/-M` stats, hunk headers, and twin old/new line-number gutters. `Edit` / `Write` / `str_replace`-style tool calls in a chat now expand into the same structured diff (synthesized from `old_string` / `new_string` via line-level LCS); fenced ` ```diff ` blocks render through the same component.
- Sequential streaming for chat responses (Claude Code style). `POST /chats/:id/messages/stream` emits `{tool_call: {name, input, summary}}` and `{tool_result: {name, output, summary}}` SSE frames inline with `{content}` and `{thinking}`, in the order the agent produced them. Extended-thinking blocks are streamed and persisted as `chat_messages` rows with `role: "thinking"` that render as collapsible "Thought for …" blocks above the assistant reply. GitHub Copilot chats surface tool events through the same path so Copilot and Claude Code render identically on both live stream and reload.
- Chat attachments accept PDFs and any UTF-8 text-like file (source code, CSV, XML, JSON, YAML, Markdown, HTML, logs, diffs, …) in addition to images. Content type is determined by magic-byte sniff + filename extension rather than the browser-supplied MIME. The backend forwards PDFs as Anthropic `document` base64 blocks and text files as `document` plain-text blocks (truncated at 1 MB with a `[truncated N bytes]` marker) alongside `image` blocks. Spoofed uploads (client claims an image but content is not) are rejected with 422. Per-attachment cap raised from 10 MB to 30 MB; per-message total cap raised from 20 MB to 30 MB, matching the Claude Code CLI.

The widened `MessageContent = string | AnthropicContentBlock[]` flows through `priorMessages` and the current prompt without a parallel multimodal path. `POST /chats/:id/messages` and `/messages/stream` accept `attachment_ids: number[]` (max 10), validate each id belongs to the chat and is still unlinked, enforce a 30 MB total size cap, and populate `chat_attachments.message_id` atomically. `GET /chats/:id/attachments/:attId/blob` streams the stored file inline with the saved `mime_type`, `Content-Length`, `X-Content-Type-Options: nosniff`, and `Content-Disposition: inline; filename*=UTF-8''…`. Cross-chat access returns `404`. The chat transcript renders attachments as a thumbnail grid below the bubble with a local click-to-open lightbox.

The dialog pre-fills title / symptom / root cause / resolution / tags from the Haiku extractor; the tags input offers a typeahead. Save posts to `POST /incidents` with `createdByChatId` set to the originating chat.

Degrades to an empty draft when no active AI keys are configured, the LLM call fails, or `skipExtract: true` is passed — never surfaces 5xx on extractor failure.

- Past-incident retrieval at agent-session start. The first turn's system prompt is augmented with a compact `<past_incidents>` block listing up to 5 incidents whose symptom / root cause / resolution best match the initial user prompt (scoped to the task / chat's project when set). Logs `incidents.injected count=N ref=…` per run.
- Default skills seeder. On every daemon boot, each skill under `src/resources/default-skills/` (currently `state-machine-driven-task`) is copied into `~/flockctl/skills/` if the target directory is absent. Copies — not symlinks — so users may freely edit; presence of the destination directory is the "already-seeded" marker.
- `flockctl state-machines check [--diff=<ref>] [--files=<glob>]` CLI — pre-commit validator that compares a git diff against the project state-machine registry (`.flockctl/state-machines/*.md`, Mermaid `stateDiagram-v2`). Detects new transitions via explicit `// @sm:<entity> <from> -> <to>` annotations, `{ from: "X", to: "Y" }` object literals, and unknown `.transition("event")` calls; exits `1` with a `file:line  new transition X→Y not declared in registry` message plus a suggested mermaid edit, or `0` when every detected transition is declared. `// flockctl-sm-ignore` on the same / previous line suppresses a false positive.
- `waiting_for_input` task status. When an agent calls `AskUserQuestion`, the task is parked instead of consuming a concurrency slot or accruing tokens. Answering restores `running`; if the daemon restarted while the task was parked, answering flips it back to `queued` and the next `execute()` call resumes the prior Claude Code session via `claudeSessionId`. Parked tasks are intentionally skipped by `resetStaleTasks()` at boot. `GET /tasks/:id/pending-questions`, `POST /tasks/:id/question/:requestId/answer`, and the chat counterparts `GET /chats/:id/pending-questions` / `POST /chats/:id/question/:requestId/answer` close the loop. The `/answer` variant distinguishes unknown / wrong-owner requestId (404) from already-resolved questions (409), caps the answer at 1–8000 chars (400 on oversize), and echoes the post-answer task status.
- `GET /attention` endpoint and `attention_changed` WebSocket event surfacing every task or chat blocked on a user decision (permission, approval, agent question).
- Project / workspace TODO.md. Seeded at the root of every newly created project / workspace (`POST /projects`, `POST /workspaces`, `POST /workspaces/:id/projects`); existing TODO.md files are never overwritten. A "TODO" button on detail pages opens a Markdown viewer with an inline editor; persistence via `GET/PUT /projects/:id/todo` and `GET/PUT /workspaces/:id/todo` (256 KB cap, identical to AGENTS.md).

Each honours `FLOCKCTL_HOST`, `FLOCKCTL_PORT`, `FLOCKCTL_TOKEN`, accepts numeric id or case-insensitive name for `<idOrName>`, and supports `-k, --allowed-key-ids <ids>` (comma-separated) on project / workspace creation to satisfy the new mandatory allow-list.

- "Browse…" directory picker in the Create Project / Workspace dialogs. Opens the DirectoryPicker modal and writes the selected directory back into the input; the text input itself remains editable (paste / manual entry still work — the only option in remote mode where `/fs/browse` is loopback-only). Single-clicking an entry highlights it; "Will select:" preview line shows the exact path that will be returned.

The flat two-section list is now four labelled groups — Overview, Work, Automate, System — each with a chevron toggle and rollup count badge summarising attention-worthy items. Per-group collapsed state is persisted in `localStorage` (`flockctl_sidebar_collapsed`).

### Changed
- AGENTS.md is human-owned. Flockctl no longer regenerates `<project>/AGENTS.md`; it merges three layers (user, workspace-public, project-public) in memory and injects the result into the agent system prompt at session start. Project and workspace settings pages show a single flat `AGENTS.md` editor with an effective-preview accordion instead of the old public/private tabs.
- Root `AGENTS.md` and `CLAUDE.md` symlink are always generated in every project and workspace directory, even when the editable source is empty. In that case the root `AGENTS.md` contains only the auto-generated header explaining where to edit. Previously both files were removed when the source was empty.
- Tasks page "Failed" stat card counts only still-broken failures — failures whose re-run already succeeded fold into Completed. Tasks table replaces the "Agent" column with an "AI Key" column showing the assigned key label, and the "Model" column shows the model the provider actually used (falling back to the requested model, then "Default"). Board-view slice detail panel shows a "Run" label for slices in `pending` / `planning` status instead of "Re-run" so first-time execution is not mislabeled as a retry. Completed milestones and slices render a green check icon in the Board-view tree panel and milestone detail list. The "Re-run" button is no longer shown on `done` tasks (it only appears for `failed` and `timed_out`).
- Tasks UI surfaces the model used for each task: a "Model" column on the tasks list (the requested model, or "Default" when unset) and a "Model" field on the task detail page that prefers models actually recorded in `usage_records`.
- Plan-entity chat dialog (milestone / slice / task) streams through the unified `POST /chats/:id/messages/stream` endpoint instead of the legacy `POST /projects/:pid/plan-chat/stream`. The dialog resolves (or lazily creates) the entity-scoped chat via `POST /chats` and restores persisted messages from cache when reopened. Plan-entity chat dialog and workspace chat panel share the full chat page's feature set via the new `ChatConversation` component (model / AI-key / permission-mode pickers, file attachments, pending-permission cards, TodoWrite progress + history, agent-question prompts, retry-on-missing-response, Save as incident multi-select). Workspace chats now persist messages server-side.
- Workspace chats and workspace-scoped tasks see the full list of projects in their workspace as part of the agent's system prompt (name, absolute path, description, scoping rules). Task sessions pick this up via a new `workspaceContext` option on `AgentSession`; chat sessions get the same block both via the workspace-aware system prompt and via session-level injection so explicit `body.system` overrides no longer strip the project list.
- `PATCH /chats/:id` with `permission_mode` propagates the new mode to the in-flight chat session instead of only taking effect on the next turn. Relaxing the mode (e.g. `default` → `bypassPermissions` / `acceptEdits`) auto-resolves every pending request the new mode would have allowed. Switching to `auto` re-runs the path-scoped decision per pending request. New `chat_permission_mode_changed` WebSocket event broadcasts the transition; bulk-auto-resolved requests emit canonical `permission_resolved` frames.
- Chat transcript visual refresh: softer `rounded-2xl` bubbles with subtle border and shadow for the assistant, consistent radius family across tool strips and error banners, more breathing room between messages, and a proper typography scale (headings, lists, blockquotes, horizontal rules) for markdown inside assistant bubbles.
- `POST /chats` is idempotent when `projectId` + `entityType` + `entityId` are all supplied — returns the existing chat row (HTTP 200) instead of creating a duplicate. Without the full triple, behaviour is unchanged (HTTP 201). Composite index `idx_chats_entity (project_id, entity_type, entity_id)` so entity-scoped lookups are O(log n).
- Task concurrency is enforced per AI Provider Key (default 5 concurrent tasks per key) instead of a single process-wide cap. Scheduler fallback skips saturated higher-priority keys and dispatches onto the next available key.
- `PUT /tasks/:id` with `{state: "ready"}` refuses the transition (`400 {error: "spec_required"}`) when the task's plan has `spec_required: true` in its milestone YAML frontmatter and `acceptance_criteria` is empty. New milestones default to `spec_required: true`; legacy milestones default to `false`. Authoring-only — the task executor does not inspect the flag.
- `GET /health` reads the version from the bundled `package.json` instead of `npm_package_version` (which was unset when launched via the `flockctl` bin).
- `make reinstall` refuses to proceed if port 52077 is still busy after `flockctl stop` instead of force-killing whatever is on the port — the operator is expected to investigate the stuck process.
- Dashboard, analytics, project, and workspace charts share a single `chart-theme.ts` module — tooltip, grid, tick, and legend styles are no longer duplicated per page; every tooltip uses `var(--popover-foreground)` text on `var(--popover)` background so it stays legible in both light and dark themes.

### Fixed
- Chat context no longer resets when restarting the daemon (`flockctl stop && flockctl start`, `make reinstall`, tsx watch reloads). `flockctl stop` blocks until the daemon actually exits (up to 15 s) and the shutdown path waits the same 15 s for in-flight chat streams to finish, so each chat's Claude Code `session_id` flushes to SQLite before the process dies.
- WebSocket streams (tasks, chats, global attention) close and reopen against the new backend URL when the active server is switched in the sidebar — previously the socket stayed pointed at the old server until a full page reload.
- Flockctl agent now sees AGENTS.md content in its system prompt (the Claude Agent SDK path previously did not inject it).
- Final assistant message no longer renders twice at the end of a chat turn. The live transcript is hidden as soon as both the local SSE stream and the server session have ended, so the persisted assistant row stops rendering alongside residual live blocks from the same turn.
- Agentic chat responses render as one message per turn instead of one giant merged blob — for both Claude Code and GitHub Copilot. `POST /chats/:id/messages/stream` flushes the accumulated assistant text as its own row at every `tool_call` boundary AND at every `turn_end` event. Each assistant turn maps 1:1 to a persisted row.
- Sending a second message while a previous turn was still streaming could clobber the in-flight fetch's abort controller. `useChatStream` now aborts any leftover fetch before replacing the controller, and the composer routes mid-stream submissions through the queue so only one stream per chat is ever active.
- Stop button in the chat composer aborts the in-flight agent session on the backend (previously it only closed the client-side SSE, leaving the AgentSession running and consuming tokens). `cancelStream()` POSTs to `/chats/:id/cancel` so `chatExecutor.cancel()` fires `session.abort()`. Stop button also survives a page reload while a chat turn is in flight — the composer renders Stop whenever either the local SSE reader is active OR the backend reports the session is still running.
- Chat UI no longer shows a spurious "Response was not received" bubble after switching to another chat and back while the agent is still streaming. The optimistic user-message update flags the chat as running in the cache, the event-stream hook force-refetches the chat on (re)subscribe, and the `session_ended` handler refetches the chat detail BEFORE flipping `sessionRunning` to `false`, closing the race that briefly left `serverRunning=false` while the optimistic cache still ended with the user message.
- `POST /chats/:id/messages` (non-streaming) now resolves its system prompt through the same `resolveChatSystemPrompt` pipeline as `/messages/stream`. Previously it fell back to `"You are a helpful AI assistant."`, silently stripping workspace + entity context for any caller hitting this endpoint directly.
- `POST /chats/:id/messages` and `/messages/stream` return `422` with a human-readable message when the resolved `keyId` is not in the chat's project/workspace `allowedKeyIds` whitelist — naming the rejected key, listing allowed keys by label + provider, and hinting which knob to update. Previously the chat would silently run under a different provider.
- Chat composer auto-select waits for the project's `allowedKeys` fetch to resolve before picking a key, and re-picks if the allow-list tightens out the current selection mid-session. `POST /chats` validates a caller-supplied `aiProviderKeyId` against the project's allow-list (422 on disallowed) and, when no key is supplied for a project with an allow-list, auto-fills a permitted key instead of persisting `NULL`.
- Chat transcript no longer shows a column of empty gray bubbles from tool-only or interrupted assistant turns. The backend still persists the turn for continuity, but the UI suppresses an assistant bubble when its content is empty/whitespace and it has no attachments. Tool-call / tool-result rows (persisted with `role: "tool"`) render as compact one-line tool strips with an expandable input/output pane (the response layer auto-parses JSON-looking string fields, so `StoredToolMessageItem` accepts both raw string and already-parsed object).
- Pending chat attachments survive chat navigation (composer chip list moved to a per-chat module store).
- Unrelated files no longer leak into task / chat diffs. Both `GET /tasks/:id/diff` and the new `GET /chats/:id/diff` are derived from an in-DB file-edit journal populated from the agent's `Edit` / `Write` / `MultiEdit` / `str_replace` tool inputs, instead of running `git diff` against the shared working tree. Parallel tasks or chats no longer cross-contaminate; pre-existing working-tree dirt is ignored. Bash-driven file changes are outside the journal — same blind spot as Cursor / Copilot.
- GitHub Copilot chats run in the chat's project directory instead of the daemon's `process.cwd()`. `CopilotProvider` forwards the chat's resolved cwd as `workingDirectory` on each `client.createSession(...)` call, which the Copilot SDK uses to scope tool operations and cwd-derived context.
- Usage-record pricing follows the attached AI Provider Key's `provider`, not a model-name guess. A Copilot-backed task on `claude-opus-4.7` is now correctly priced $0 (flat-rate) and stamped `usage_records.provider = "github_copilot"`. Chat-list cost indicator surfaces again for Claude Code CLI ($-equivalent surrogate) and GitHub Copilot (`N PR` premium-request quota) where it had been blank because both priced at $0.
- Daemon startup no longer crashes when a queued task references a missing `prompt_file`. `TaskExecutor` treats prompt-resolution failures as task-level failures (`status=failed` with `errorMessage`) and keeps the backend alive.
- Queued tasks left behind in the in-memory queue when the daemon shuts down are adopted on the next boot. `resetStaleTasks()` widened to include `queued` rows. Auto-executor no longer spawns a duplicate execution task for a plan task that already has a non-terminal exec task attached — re-triggering reuses the existing `execution_task_id` instead of leaving zombie queued rows. Startup sweeps `cancelOrphanedExecutionTasks()` before re-queueing, cancelling non-terminal exec tasks whose plan file has since repointed to a newer `execution_task_id`.
- Task detail page shows the `prompt_file` path when a task was created from a plan spec (previously the "Prompt" card rendered nothing because `task.prompt` is null on plan-driven exec tasks).
- Chat UI no longer shows a spurious "Response was not received" error after sending a message, switching to another chat, and coming back mid-setup. The streaming route calls `chatExecutor.claim(id)` immediately after persisting the user row so a concurrent `GET /chats/:id` still reports `isRunning=true`.
- Globally-registered MCP servers (`~/flockctl/mcp/*.json`) are visible to agents spawned via the Claude Agent SDK. The SDK does not auto-read the project's `.mcp.json` — only the interactive `claude` CLI does — so `AgentSession` resolves MCP servers once per run (project > workspace > global precedence) and forwards them through `provider.chat({ mcpServers })` → `sdk.query({ options: { mcpServers } })`. Before this fix, any `mcp__<name>__*` tool call failed with "tool not found" inside tasks and chats.
- Directory picker no longer silently returns the parent directory when the user single-clicks an entry and then presses "Select". Clicking or arrowing onto an entry highlights it; "Select" returns the absolute path of that highlighted entry; with nothing highlighted it still returns the directory currently being viewed.

### Removed
- Tasks page "Cards" view (`?view=cards`) — the cross-project Kanban replaces it. Saved bookmarks pointing at `?view=cards` silently degrade to the Table view.
- Workspace→project AGENTS.md cascade (`reconcileAllProjectsInWorkspaceAgents`). Editing workspace guidance no longer modifies project files.
- Milestone execution dashboard page (`/projects/:projectId/execution/:milestoneId`) and the "Dashboard" button that linked to it from each milestone row.
- "Add Slice" button on milestone rows and "Add Task" button on slice rows. Slices and tasks are authored on disk under `.flockctl/` and picked up by the plan tree.
- `POST /projects/:pid/plan-chat/stream` endpoint. The plan-entity chat dialog uses the unified `POST /chats/:id/messages/stream` flow exclusively; no callers, no redirect.
- `task_templates` DB table and the numeric template id. `assignedKeyId` removed from templates (moved onto schedules).
- Task-level `stateMachine` spec field. The column is dropped from `tasks` via migration `0035_drop_task_state_machine.sql`; `POST /tasks` and `PUT /tasks/:id` no longer accept `stateMachine` in the request body, and `GET /tasks/:id` no longer returns it. State-machine contracts now live exclusively in the project-level registry under `.flockctl/state-machines/*.md`.
- YAML state-machine DSL (`parseStateMachine`). Mermaid `stateDiagram-v2` inside the registry markdown is now the only supported authoring format; the `js-yaml` frontmatter loader stays for registry parsing.
- State-machine Vitest harness generator (`generateTestsFromSM` / `sm-test-generator.ts`). Transitions declared in the registry are covered by hand-written tests; the `flockctl state-machines check` gate catches undeclared transitions in code.
- `transition` trigger on the `state-machine-driven-task` skill — too broad. The skill now activates only on the literal phrase "state machine".

### Security
- Agents can no longer read or write `.mcp.json` (the reconciled MCP config containing plaintext secret values), the master encryption key `~/Flockctl/secret.key`, or the SQLite database `~/Flockctl/flockctl.db` (and its WAL/SHM sidecars). A hard path denylist runs inside `canUseTool` for every permission mode — including `bypassPermissions`, `acceptEdits`, and `plan` — and returns a deny decision before mode-specific allow shortcuts fire. MCP configs should be edited via the source files under `.flockctl/mcp/` (which contain `${secret:NAME}` placeholders, not real values); the reconciler regenerates `.mcp.json` automatically. `Bash` is not covered by the denylist — shell-based exfiltration is a known gap.
- `node-notifier` transitive dependency bumped so the vulnerable `uuid@3` path is gone from the install tree (`npm audit` reports 0 issues in both root and `ui/`).
