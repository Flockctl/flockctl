# Security model

This document describes Flockctl's threat model and the guarantees it provides in each mode. For **how to turn remote access on** see [REMOTE-ACCESS.md](REMOTE-ACCESS.md); for **what each config field does** see [CONFIGURATION.md](CONFIGURATION.md).

## Operating modes

| Mode | Trigger | Listener | AuthN | CORS |
|------|---------|----------|-------|------|
| **Local** (default) | `flockctl start` / `--host 127.0.0.1` | `127.0.0.1:52077` only | none | wildcard |
| **Remote** (opt-in) | `flockctl start --host <non-loopback>` with token configured | whatever host you named | Bearer token(s) — localhost bypass still applies | whitelist from `corsOrigins` |
| **Remote, unauth** (escape hatch) | `flockctl start --host <non-loopback> --allow-insecure-public` | whatever host you named | none (prints `[SECURITY WARNING]`) | wildcard |

**Key invariant:** the server refuses to start on a non-loopback interface without either a configured token or the explicit `--allow-insecure-public` flag. Forgetting to configure auth no longer silently exposes the daemon.

## Threat model

### Local mode

**Trust boundary:** anyone with a process running as your user on your machine.

**Assumptions:**

- Attackers cannot speak to `127.0.0.1:52077` unless they already have local code execution as your user.
- Any other process owned by the user can call every endpoint — identical trust boundary to a standard user-owned daemon.

**Out-of-scope for local mode:**

- Forwarding local mode to the network (nginx port-forward, `kubectl port-forward`, tailnet ACL…). Local mode has no auth; if you need network exposure, switch to remote mode — do **not** front local mode with anything that exposes beyond loopback.

### Remote mode

**Trust boundary:** anyone holding one of the configured `remoteAccessTokens`.

**Guarantees (enforced by `src/middleware/remote-auth.ts`):**

| Control | Mechanism |
|---------|-----------|
| **Localhost bypass** | Requests whose socket remote address is `127.0.0.1`, `::1`, or `::ffff:127.0.0.1` skip auth. IP comes from the Node socket, not from `X-Forwarded-For` / `Forwarded` — header spoofing can't forge "localhost" |
| **Bearer token** | Non-local HTTP requests must send `Authorization: Bearer <token>`. Matched against every configured label |
| **Timing-safe compare** | `crypto.timingSafeEqual` — the loop runs to completion to defeat early-exit timing leaks |
| **Minimum token length** | Entries < 32 chars are dropped at load time with a `[SECURITY]` warning; the daemon behaves as if no token were set — no false-secure configuration |
| **WebSocket auth** | Non-local WS connections must include `?token=<token>` (browsers can't set headers on `new WebSocket(url)`); failure closes with code `1008` |
| **Rate limit** | 5 failed auth attempts per IP per 60 s → `429`; counter resets on success or window expiry |
| **CORS whitelist** | Only origins in `corsOrigins` are allowed. Empty / absent → wildcard (treat that as misconfiguration for remote mode) |
| **Public paths** | `GET /health` and `OPTIONS` preflight always pass — for probes and CORS |
| **`.flockctlrc` permissions** | Startup warns if the file isn't mode `600` while a token is active. `saveRc()` re-applies `600` after every write |

**Token properties:**

- 32 bytes of entropy, base64url — brute-forcing is infeasible even without the rate limiter.
- Labeled: one per device / purpose. Revoking a leaked laptop token does not invalidate the phone or tablet tokens.
- Generated by `flockctl token generate` using `crypto.randomBytes(32)`.

**What remote mode does NOT provide:**

- **No per-user accounts, no RBAC.** Every valid token has full API access. Flockctl is designed for **personal** remote access (you ↔ your own server from your own devices), not multi-tenant team use. For team scenarios, front the daemon with a reverse proxy that handles SSO and per-user authorization.
- **No audit log persistence.** The middleware stashes the matching token's label on the Hono context (`authTokenLabel`) so future hooks can log it, but nothing is written to disk today.
- **No TLS termination.** Flockctl speaks plain HTTP. Put it behind a TLS-terminating reverse proxy.
- **No mTLS, no JWT, no session cookies.**

### Remote, unauth (break-glass)

Starting with `--allow-insecure-public` is equivalent to local mode's openness, but exposed on the network. Anyone who can reach the address can run Claude Code on your machine with your API keys.

**Only acceptable** when the daemon is on a trusted VPN / tailnet segment already gated by something else (Tailscale ACLs, firewall, private mesh). Prints `[SECURITY WARNING]` on every startup.

## Threat matrix

| Threat | Local mode | Remote mode |
|--------|-----------|-------------|
| Unauthenticated attacker on the network | N/A — loopback only | Blocked by Bearer token + CORS |
| Credential stuffing against the token | N/A | Rate limiter caps to 5 attempts per IP per 60 s; 32-byte entropy makes brute force infeasible |
| CSRF from a malicious origin | Wildcard CORS, but loopback-only — an attacker's browser can't route to `127.0.0.1` without SSRF | CORS whitelist blocks non-whitelisted origins; Bearer header is not auto-sent by browsers |
| API key leakage | Keys stored plain in SQLite; `GET /keys` redacts values; DB file should be user-only | Same, plus Bearer token guards `/keys` |
| Claude CLI credentials | Stored in `~/.claude*`; Flockctl reads, never modifies | Same |
| Arbitrary code execution via agent tools | Expected — agents need filesystem access. Scope via `permissionMode` and project paths | Same — a compromised token yields the same capability as local physical access |
| SQL injection | Drizzle ORM uses parameterized queries everywhere | Same |
| Path traversal | Plan store slugifies paths; working dirs validated against project paths | Same |
| Header spoofing (fake localhost) | N/A | Blocked — IP read from socket, not headers |
| Short/weak token | N/A | Rejected at load time with `[SECURITY]` warning |
| World-readable `~/.flockctlrc` | Harmless (no secrets) | Token leaks to any local user → warning on boot; `saveRc()` enforces `600` on writes |

## API key handling

Keys live in the `ai_provider_keys` table as plaintext — acceptable for a user-owned local daemon, **not** a shared secret store. Don't reuse the DB file across trust boundaries.

### Anthropic

- `sk-ant-api*` → `x-api-key` header (classic API keys)
- `sk-ant-oat*` → `Authorization: Bearer` + `anthropic-beta: oauth-2025-04-20`
- Raw OAuth access tokens (no `sk-ant-` prefix) → same as above

### Redaction

- `GET /keys` redacts `key_value` in responses.
- `GET /meta/remote-servers` redacts `token` (`hasToken: true|false`).

### Rotation & failover

- Keys have a priority and an error-backoff counter.
- Flockctl rotates away from failing keys automatically — no manual intervention on rate limits.

### Scoping (`allowedKeyIds`)

- Set on a **workspace** to restrict which keys its projects may use.
- Set on a **project** to override the workspace default.
- Unset at both levels → every active key is eligible.

## Recommendations

1. **Default to local mode.** If you can reach the daemon from `127.0.0.1`, stay there — the server enforces this by default.
2. **Before exposing the daemon beyond loopback** — generate a labeled token with `flockctl token generate --save`, set `corsOrigins`, and put a TLS-terminating reverse proxy in front. Do not skip any step. `--allow-insecure-public` is a break-glass flag, not a shortcut.
3. **One label per device.** Generate a separate token for each phone / laptop / tablet / CI runner. Revoke leaked ones individually with `flockctl token revoke <label>`.
4. **Protect `~/.flockctlrc`** — `chmod 600 ~/.flockctlrc`. Flockctl re-applies this on every write, but existing files aren't retroactively tightened.
5. **Protect `~/flockctl/flockctl.db`** — user-only permissions; contains API keys and task history.
6. **Use separate Claude CLI config dirs** for different accounts to isolate credentials — see [MULTI-ACCOUNT.md](MULTI-ACCOUNT.md).
7. **Audit `permissionMode` per project** — it gates how freely agents may invoke destructive tools (`Bash`, `Write`, `Edit`). `bypassPermissions` should only be used for trusted automation.
